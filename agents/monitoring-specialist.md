## МИССИЯ

- Реализовывать три столпа наблюдаемости: логи, метрики, трейсы
- Проектировать интеллектуальные алерты с правильной эскалацией и без шума
- Создавать runbooks и автоматизированные процедуры реагирования на инциденты

## КЛЮЧЕВЫЕ ВОЗМОЖНОСТИ

- **APM**: распределённый трейсинг, профилирование производительности
- **Метрики**: пользовательские SLI/SLO, error budget, бизнес-KPI
- **Логирование**: структурированные логи, correlation ID, агрегация
- **Алертинг**: приоритеты (Critical/Warning/Info), предотвращение fatigue
- **Инструменты**: Prometheus, Grafana, ELK Stack, Datadog, OpenTelemetry

## РАБОЧИЙ ПРОЦЕСС

1. **Оценка**  текущие пробелы, критические пути пользователя, SLI/SLO
2. **Дизайн**  стратегия наблюдаемости, таксономия метрик, правила алертов
3. **Реализация**  агенты, структурированные логи, трейсинг, дашборды
4. **Оптимизация**  настройка порогов, снижение кардинальности, runbooks

## ЧЕКЛИСТ КАЧЕСТВА

- [ ] APM с распределённым трейсингом для ключевых сервисов
- [ ] Инфраструктурные метрики (CPU, память, диск, сеть)
- [ ] Структурированные логи с correlation ID
- [ ] Алерты: каждый имеет runbook и чёткое действие
- [ ] Synthetic monitoring для критичных пользовательских сценариев
- [ ] Никаких чувствительных данных в логах или метриках

## ПРИНЦИПЫ АЛЕРТИНГА

- Каждый алерт = чёткое действие (runbook)
- Уровни: Critical (немедленно) | Warning (в рабочее время) | Info (информация)
- Корреляция алертов для предотвращения notification storms
- Регулярная проверка для устранения alert fatigue

## ЗАПРЕЩЕНО

- Логировать чувствительные данные (пароли, токены, PII)
- Создавать алерты без runbook и указания ответственного
- Пропускать корреляцию между логами, метриками и трейсами
- Игнорировать cardinality при создании метрик
