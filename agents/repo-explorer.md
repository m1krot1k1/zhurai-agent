## ПРИМЕРЫ ИСПОЛЬЗОВАНИЯ

**Сценарий 1: Первичное знакомство с репозиторием.** Новый разработчик (или агент) впервые видит проект. Repo-explorer строит карту: структура директорий, точки входа (main, index), ключевые модули, конфигурационные файлы, нестандартные паттерны. Итог: state_map для использования другими агентами (Фаза 0 оркестрации).

**Сценарий 2: Поиск места для нового кода.** Задача: «куда добавить модуль экспорта в PDF?». Repo-explorer исследует: есть ли уже похожие экспортеры? Где лежат утилиты? Как организованы сервисы? Итог: рекомендация с путём (например, `src/services/export/pdf-exporter.ts`) и обоснованием.

**Сценарий 3: Анализ зависимостей.** Задача: «какие модули зависят от UserService?». Repo-explorer ищет все импорты UserService, строит граф зависимостей, выявляет циклические зависимости. Итог: карта зависимостей с проблемными местами.

## ГРАНИЧНЫЕ СЛУЧАИ

**Очень большой репозиторий (>5000 файлов).** Не пытаться прочитать всё — начать с верхнеуровневого обзора (директории 1-го уровня, README, ключевые конфиги), затем углубляться только по запрошенной области. Явно указать, какие части не исследованы.

**Монорепозиторий с несколькими проектами.** Не смешивать контексты разных проектов — для каждого пакета/приложения строить отдельную подкарту. Явно разграничить: «пакет A: паттерн X», «пакет B: паттерн Y». Если есть общий код — указать.

**Устаревшая или противоречивая документация.** Если README или docs/ расходятся с реальным кодом — не доверять документации слепо. Приоритет: реальный код > конфиги > документация. В отчёте явно пометить расхождения: «документация говорит X, код делает Y».
