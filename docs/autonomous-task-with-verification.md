# Start Supervisor → Execute → Verify → Analyze → Rework

Короткий контракт для цикла с **разделением ролей** (согласован с [repo-task-proof-loop skill](../skills/repo-task-proof-loop/SKILL.md)).

## Роли

| Фаза | Кто | Действие |
|------|-----|----------|
| Supervise | Корневой **`start`** | Запускает волну через **`Task(orchestrator)`** напрямую, анализирует `START_REPORT`, решает `continue/stop` |
| Execute | `orchestrator` + специалисты (`code`, `test-specialist`, …) | Реализация по зафиксированной спецификации внутри одной волны |
| Verify | Отдельный проход (`test-specialist`, `review`, ручные проверки) | Сверка с AC, тесты, линт — **без правок прод-кода** при чистой верификации |
| Analyze | Корневой **`start`** или optional `start` в `ENTRY_MODE: analyzer_only` | Сравнение артефактов с AC / `START_REPORT` → «Выполнено» или «Продолжить» + список зазоров |
| Rework | Следующая волна: **`Task(orchestrator)`** → нужные специалисты | Минимальные исправления по зазорам, затем снова Verify |

## Важно для `orchestrator`

- Канонический многоходовый цикл идёт через **root `start` → `Task(orchestrator)` напрямую**. Корневой `start` сам **не** открывает ветки к специалистам и **не** читает repo до handoff.
- **Устарело:** вызов **`Task(subagent_type="start", …)`** из корневого `start` как **`ENTRY_MODE: supervised_worker`** (worker-start) — раньше использовался как промежуточный hop; **удалён из цепочки**, потому что активный канон теперь flat-chain `root start -> Task(orchestrator)` без лишней промежуточной роли. Вложенные subagent'ы при этом по-прежнему могут использовать `Task()` там, где это разрешают их собственные правила.
- Вызов **`Task(subagent_type="start", …)`** из оркестратора допустим **только** как **`ENTRY_MODE: analyzer_only`**: вход — текущие доставки и AC, выход — вердикт «готово / не готово», **без** новой маршрутизации и **без** `Task(start)` внутри этого вызова.
- `start` в режиме **analyzer_only** не открывает `Task(start)` дальше — отчёт всегда возвращается наверх.

## Анти-петля

Повторять цикл только при **новых артефактах или новой проверке** (см. `rules/orchestrator.mdc`). Одинаковый «отпечаток» задачи без прогресса → смена стратегии или эскалация пользователю.

## Многоитерационное самоулучшение этого репозитория

Если цель — **несколько кругов** улучшения конфигурации агентов с прогоном тестов и анализом без участия пользователя на каждом шаге, используйте каноническую спираль: [autonomous-self-improvement-loop.md](./autonomous-self-improvement-loop.md) (root `start` владеет outer-loop и запускает **`Task(orchestrator)`** на волну; `orchestrator` исполняет волну).
