# Процесс и quality gates

## Маршрутизация

- **`/start`** → только **`orchestrator`** (один `Task`); специалисты — из оркестратора. Итог пользователю формулирует **start** после ответа оркестратора.
- **Прямой `/code`, `/debug`, …** → соответствующий специалист без start.
- **Исполнимый запрос** (императив, DUA) → работа без блокировки на PBI/task.
- **Только объяснение** → оркестратор может отдать ветку `ask` или одну ветку-специалиста; при прямом вызове — `/ask`.
- **Мульти-артефакт / мульти-домен** → `orchestrator` с явными ветками и владельцами.

## Минимальные проверки

- Код: компиляция / тесты по возможности; иначе явно указать «не запускалось: причина».
- UI: дымовая проверка в браузере при наличии инструментов; иначе явно отметить пропуск.
- Завершение ветки: completion contract (что проверено).
- При изменениях в `agents/*.md`: сначала проверка количества `scripts/validate-agent-registry.sh` (bash) / `scripts/validate-agent-registry.ps1` (PowerShell), затем проверка согласованности `scripts/validate-repo-consistency.sh` / `scripts/validate-repo-consistency.ps1`.

## Final Verification Checklist

Перед финальным ответом проверь:

- [ ] Все Acceptance Criteria покрыты явно, без «подразумевается».
- [ ] Есть список изменённых путей.
- [ ] Проверки действительно запускались (или явно указано, что не запускалось и почему).
- [ ] Отдельно отмечены остаточные риски/ограничения.
- [ ] Completion contract заполнен в одном формате.

Минимальный шаблон completion contract:

```yaml
summary: <что сделано>
files_changed:
  - <path>
checks_run:
  - name: <check name>
    result: pass|fail|not_run
    evidence: <короткое доказательство>
acceptance_criteria:
  - criterion: <текст>
    status: met|not_met
residual_risks:
  - none|<риск>
confidence: high|medium|low
```

## Крупные задачи

- Skill **repo-task-proof-loop**: канонические артефакты в `agent-tasks/<TASK_ID>/`; в установленном проекте runtime-путь будет `.cursor/agent-tasks/<TASK_ID>/`. Базовый набор: `spec.md`, независимый `verdict.md`.

## Автономность и ответ пользователю

- По умолчанию — действовать с разумными допущениями; вопросы только при блокирующем риске или нехватке секретов.
- Футер ответа: см. `skills/start-workflow/SKILL.md` (краткая сводка, векторы улучшения, опционально `+` — автономное исполнение и формат ответа теперь в `start-workflow`).
